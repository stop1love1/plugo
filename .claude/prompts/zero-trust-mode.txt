## Operating Rules

- Evidence-based only: every finding MUST include file path + line range + exact snippet.
- Distinguish production code vs dev/test-only — flag both but label clearly.
- Do NOT flag standard library usage unless the specific call context makes it exploitable.
- For lockfiles: check for git+/http URLs, unusual registries, postinstall scripts. Delegate deep CVE scanning to dedicated tools (npm audit, pip-audit, trivy).
- Highlight both: (a) what IS dangerous now, and (b) what BECOMES dangerous if env vars / user inputs are attacker-controlled.

## Severity Classification

Use these levels consistently. Do not inflate severity.

- CRITICAL: Active exploitation possible with no/low barrier. Data breach, RCE, or full compromise.
- HIGH: Exploitable with moderate effort or specific conditions. Significant impact.
- MEDIUM: Requires chained conditions or insider access. Limited blast radius.
- LOW: Theoretical risk, defense-in-depth concern, best practice violation.
- INFO: Observation or recommendation with no direct exploitability.

For each finding, state:
- Exploitability: trivial | moderate | complex
- Impact: data breach | RCE | privilege escalation | DoS | info disclosure
- Component: production | dev-only | build-time

## Audit Protocol (Execute in order)

### Phase 1: DEPENDENCY FORENSICS

Inspect dependency manifests (package.json, requirements.txt, pyproject.toml):
- Typosquatting / look-alike package names (e.g. reqests vs requests)
- Obscure, abandoned, or unnecessarily broad packages
- Suspicious version pinning: very old versions, forks, git+ URLs, direct tarball URLs
- Lifecycle scripts that execute code: postinstall, preinstall, prepare
- For each flag: explain risk + propose safer alternative

For lockfiles (pnpm-lock.yaml): spot-check for non-registry URLs, integrity hash anomalies. Do NOT attempt full CVE cross-reference — recommend `npm audit` / `pip-audit` / `trivy` instead.

### Phase 2: MALICIOUS EXECUTION & OBFUSCATION

Search for hidden execution paths:
- Python: eval, exec, os.system, subprocess with shell=True, importlib, __import__
- JS: eval, Function(), child_process exec/spawn, dynamic import() of user input
- Shell: curl|bash, wget|sh, PowerShell encoded commands

Detect obfuscation and payload hiding:
- Base64/hex/rot encodings that decode to commands or URLs (not static assets like images/fonts)
- Long encoded blobs, XOR loops, "decrypt then exec" patterns
- Minified blobs in non-minified source contexts

Flag external resource fetching:
- Unknown domains/IPs, pastebins, raw gists, URL shorteners
- Code that downloads and executes, self-updates, or loads remote plugins

Identify persistence / backdoor patterns:
- Cron edits, startup scripts, systemd units, scheduled tasks
- Git hooks, npm lifecycle hooks, CI steps that run remote code

### Phase 3: SECRETS & DATA LEAK DETECTION

Hardcoded secrets:
- API keys, tokens, passwords, private keys, service credentials
- JWT secrets, encryption keys, database connection strings
- Check: .env tracked in git? .gitignore covers .env, *.pem, *.key?
- Check git history: `git log --all -p -- .env "*.key" "*.pem"` for leaked secrets

Accidental data exposure:
- Verbose logging of headers, cookies, request/response bodies
- Printing env vars, dumping full objects, stack traces containing secrets
- Error responses that leak internal paths, versions, or config

Telemetry / analytics that could exfiltrate:
- Crash reporters sending request context, user data, or tokens

### Phase 4: CI/CD, BUILD & RELEASE ATTACK SURFACE

GitHub Actions / CI pipelines:
- Third-party actions: pinned by SHA? Known publisher? Scoped permissions?
- Secret exposure: are secrets passed to steps that don't need them?
- Artifact uploads: could a poisoned artifact be consumed downstream?

Dockerfiles / docker-compose:
- Running as root? curl|bash installs? Adding remote GPG keys?
- Exposed ports beyond what's needed? Overly permissive network policies?
- Volume mounts that expose host filesystem?

Build scripts (Makefile, scripts/):
- Commands that fetch remote resources at build time
- Artifact tampering vectors

### Phase 5: APPLICATION SECURITY (OWASP)

This phase audits runtime application logic — not just supply chain.

Injection:
- SQL/NoSQL injection: user input reaching queries without parameterization
- OS command injection: user input in subprocess/exec calls
- Template injection: user input in string templates rendered server-side

Authentication & Session:
- JWT validation: algorithm confusion (none/HS256 vs RS256), expiry checks, secret strength
- Session fixation, token reuse across sites, missing revocation
- Password storage: bcrypt/argon2 with proper cost factor?

SSRF (Server-Side Request Forgery):
- Any endpoint accepting URLs from users (crawlers, webhooks, proxies)
- Can an attacker make the server fetch internal IPs (127.0.0.1, 169.254.169.254, 10.x)?
- URL validation: scheme whitelist (http/https only), hostname validation, redirect following

CORS & Headers:
- Wildcard origins with credentials? Overly permissive Access-Control-Allow-Origin?
- Missing security headers: CSP, X-Frame-Options, X-Content-Type-Options, HSTS

WebSocket Security:
- Origin validation on upgrade requests?
- Message injection: can a malicious client send crafted payloads?
- Rate limiting on WebSocket messages?

Input Validation:
- Trust boundaries: where does user input enter the system?
- Missing sanitization before database, filesystem, or LLM operations
- File upload: type validation, size limits, path traversal

Rate Limiting & DoS:
- Are rate limits enforced on auth endpoints (brute force)?
- Can a single client exhaust resources (large file uploads, long-running crawls)?

### Phase 6: DESERIALIZATION & UNSAFE PARSING (if applicable)

- Python pickle, joblib, dill — any deserialization of untrusted data
- Unsafe YAML loading (yaml.load without SafeLoader)
- JSON parsed then passed to eval/exec
- Model files (.pkl, .bin): warn if present, recommend .safetensors + hash verification

## Output Format

Produce a structured Security Audit Report:

### A) CRITICAL — Immediate action required
Findings with CRITICAL or HIGH severity.
Each entry:
```
[CRITICAL|HIGH] Title
File: path/to/file.py:10-25
Snippet: (exact code)
Risk: What an attacker can do
Exploit: Step-by-step scenario
Fix: Concrete patch or refactor
```

### B) SUSPICIOUS — Needs investigation
Findings that look wrong but lack conclusive evidence.
Each entry:
```
[MEDIUM] Title
File: path/to/file.py:10-25
Snippet: (exact code)
Why suspicious: explanation
Missing evidence: what would confirm/deny
Recommended action: investigate / monitor / refactor
```

### C) WEAKNESSES — Harden these
LOW severity findings and best practice violations.
Each entry:
```
[LOW|INFO] Title
File: path/to/file.py:10-25
Issue: description
Threat model: under what conditions this becomes exploitable
Fix: recommendation
```

### D) REMEDIATION PLAN
1. Prioritized fix list mapped to findings (CRITICAL first)
2. Concrete patches or code changes for each
3. Recommended tooling:
   - SAST: semgrep, bandit (Python), eslint-plugin-security (JS)
   - Dependency scanning: npm audit, pip-audit, trivy, grype
   - Secret scanning: trufflehog, gitleaks
   - Runtime: OWASP ZAP for DAST
4. Verification checklist:
```
For each finding:
- [ ] [Finding ID] Fix applied
- [ ] [Finding ID] Verified: (specific grep/command/test that confirms fix)
- [ ] [Finding ID] Re-scanned with (tool name) — clean
```

### E) SCAN DELEGATION
List the exact commands to run for checks Claude cannot perform:
```bash
# Dependency CVEs
npm audit --audit-level=moderate
pip-audit -r requirements.txt
trivy fs --severity HIGH,CRITICAL .

# Secret scanning
gitleaks detect --source . --verbose
trufflehog filesystem --directory . --only-verified

# Container scanning
trivy image plugo-backend:latest
trivy image plugo-dashboard:latest

# DAST (after deployment)
# zap-cli quick-scan http://localhost:8000
```
